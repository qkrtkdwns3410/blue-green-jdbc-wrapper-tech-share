DB 이사 중, 앱이 예전 주소로 가지 않게 막는 장치

Blue/Green 전환을 어렵게 만드는 건 DB 자체보다 앱의 연결이다. DB는 새 Green으로 바뀌었는데 앱은 잠깐 예전 Blue 주소를 기억할 수 있다.

1 Blue는 지금 쓰던 DB, Green은 새 DB다.
2 전환 중 앱의 주소록이 잠깐 낡을 수 있다.
3 bg 플러그인은 기다리거나 새 주소로 보내준다.
실제 발표 슬라이드 보기 상세 발표 대본 보기
01

한 장으로 보면 이렇다

이 발표에서 기억할 그림은 하나다. 앱은 DB 주소를 보고 찾아가고, 전환 중에는 그 주소가 잠깐 헷갈릴 수 있다.

기존 집 Blue DB

앱이 원래 붙어 있던 DB다. 전환 전에는 정상 목적지다.

새 집 Green DB

전환 후 앱이 사용해야 할 새 DB다.

문제 낡은 주소록

DB는 바뀌었는데 앱은 잠깐 예전 주소를 기억할 수 있다.

bg 플러그인 JDBC 앞의 교통정리 담당

위험한 순간에는 연결을 잠깐 멈추고, 필요하면 Green 쪽으로 보내준다.

결론 미리 켜야 함

전환 전에 켜둬야 Blue와 Green의 위치를 미리 알아둘 수 있다.

핵심은 “DB가 바뀌는 순간”이 아니라 “앱이 바뀐 DB를 제대로 찾아가는가”다. bg 플러그인은 이 짧은 혼란 구간을 JDBC 앞에서 줄여주는 안전장치다.
02

전환 중 앱은 어디로 가야 할지 헷갈린다

DB 이사로 비유하면, 앱은 주소록을 보고 집을 찾아간다. 그런데 이사 직후 주소록이 바로 새 주소로 바뀌지 않을 수 있다.

Application

평소처럼 JDBC로 DB에 연결하려고 한다.

1. 앱이 DB 주소를 본다
2. bg 플러그인이 지금 안전한지 확인한다
3. 기다리거나 Green으로 안내한다

Green DB

전환 후 앱이 도착해야 하는 새 목적지다.

왜 헷갈리나 1. DNS가 바로 안 바뀔 수 있음

DB 주소는 결국 IP로 바뀌어야 접속된다. 이 변환 결과가 잠깐 캐시될 수 있다.

왜 헷갈리나 2. JVM도 주소를 기억할 수 있음

운영체제나 DNS 서버만의 문제가 아니다. Java 애플리케이션도 조회 결과를 들고 있을 수 있다.

왜 헷갈리나 3. 이미 열린 연결이 남아 있을 수 있음

커넥션 풀은 DB 연결을 재사용한다. 전환 중에는 이 연결도 잠깐 정리가 필요하다.

그래서 Blue/Green을 만든 뒤 switchover 전에 잠깐 기다리라는 말이 나온다. 이 시간 동안 wrapper는 전환이 시작되기 전에 안정적인 DB 지도와 주소록을 미리 만들어둔다.
준비 1. Source/Target monitor를 둘 다 띄움

BlueGreenStatusProvider가 Blue 쪽과 Green 쪽 monitor를 각각 시작한다.

준비 2. 시작 시점 topology를 저장

collectTopology()가 writer, reader, cluster endpoint 같은 DB 자리 배치표를 모은다.

준비 3. 시작 시점 IP를 저장

collectHostIpAddresses()가 host 이름을 IP로 풀어서 전환 전 주소록을 만든다.

준비 4. Blue와 Green을 짝지음

updateCorrespondingNodes()가 Blue writer/reader와 Green writer/reader 대응표를 만든다.

준비 5. 전환 감지 기준을 잡음

시작 IP와 현재 IP를 비교해야 POST에서 DNS가 바뀌었는지, Green endpoint가 사라졌는지 알 수 있다.

결론. 기다리는 시간은 빈 시간이 아님

이때 만든 지도 덕분에 IN_PROGRESS에서는 기다리고, POST에서는 Green으로 우회할 수 있다.

03

단계는 다섯 장면만 알면 된다

어려운 phase 이름을 먼저 외우지 않아도 된다. 발표에서는 아래 다섯 장면으로 설명하면 충분하다.

CREATED 이사 전

아직 평소처럼 Blue DB를 써도 된다.

PREPARATION 준비 중

플러그인이 Blue와 Green의 위치를 미리 확인한다.

IN_PROGRESS 실제 이사 중

가장 위험한 순간이다. 새 연결과 쿼리를 잠깐 기다리게 할 수 있다.

POST 주소 정리 중

Blue 주소로 들어온 요청을 Green 쪽으로 바꿔 보낼 수 있다.

COMPLETED 이사 완료

정리가 끝난 상태다. 이제 Green을 평소 DB처럼 사용하면 된다.

앱 입장에서 보이는 현상

04

bg 플러그인이 하는 일은 세 가지다

주니어 대상 발표에서는 클래스 이름보다 이 세 가지 행동을 먼저 보여주면 된다.

1

상태를 계속 본다

앱이 아무 쿼리를 날리지 않아도 전환 상태는 바뀐다. 그래서 별도 monitor가 계속 확인한다.

2

위험하면 잠깐 멈춘다

IN_PROGRESS 중에는 새 연결이나 쿼리를 바로 보내지 않을 수 있다.

3

새 DB로 안내한다

정리 구간에서는 Blue로 가려는 연결을 Green 쪽으로 바꿔줄 수 있다.

BlueGreenStatusMonitor

감시 담당

runMonitoringLoop()에서 상태, topology, IP 정보를 반복해서 모은다.

BlueGreenStatusProvider

상황판 담당

source와 target에서 올라온 정보를 합쳐 현재 phase와 routing 규칙을 만든다.

BlueGreenConnectionPlugin

JDBC 입구 담당

connectInternal()과 execute()에서 지금 적용할 routing을 찾는다.

routing/*

행동 담당

Suspend*는 기다리게 하고, Substitute*는 다른 목적지로 보낸다.

발표에서는 클래스 이름을 다 외울 필요 없다. “감시 담당이 상태를 보고, 상황판 담당이 판단하고, JDBC 입구 담당이 기다리거나 우회시킨다”라고 말하면 된다.
05

헷갈리는 단어는 이렇게 읽으면 된다

처음 듣는 사람에게는 영어 용어가 제일 큰 장벽이다. 발표에서는 아래 쉬운 말부터 말하고, 괄호로 코드 용어를 붙이면 된다.

Wrapper

DB 드라이버 앞에 있는 중간 안내원. 앱과 DB 사이에서 연결을 도와준다.

DNS Cache

앱이 잠깐 들고 있는 주소록. 이 주소록이 늦게 바뀌면 예전 DB로 갈 수 있다.

Stale DNS

바뀌었어야 하는 주소가 아직 예전 IP를 가리키는 상태다.

Topology

DB 자리 배치표. writer, reader, Blue, Green의 위치를 알려준다.

Routing

요청을 어디로 보낼지 정하는 길 안내 규칙이다.

Connection Pool

DB 연결을 매번 새로 만들지 않고 모아뒀다가 재사용하는 공간이다.

Monitor

상태 확인 담당. 전환이 어디까지 왔는지 계속 본다.

Provider

상황판 정리 담당. 여러 상태를 합쳐 현재 판단을 만든다.

06

운영 중에는 잠깐 멈춘 것처럼 보일 수 있다

이 부분은 장애처럼 보일 수 있어서 미리 설명해두는 게 좋다.

A

커넥션 풀이 기다림

전환 중 새 연결을 바로 만들지 않고 잠깐 대기할 수 있다.

B

쿼리가 느려 보임

DB가 느린 게 아니라 플러그인이 안전을 위해 실행을 붙잡는 중일 수 있다.

C

timeout은 확인 신호

너무 오래 멈추면 전환 지연, 권한, DNS 문제를 확인해야 한다.

정상적인 switchover hold에 가까운 모습

  • 전환 시간대와 지연 시간이 겹친다.
  • 잠깐 대기 후 Green으로 정상 연결된다.
  • 일부 connect나 execute가 같은 구간에서 느려진다.

장애를 의심해야 하는 모습

  • timeout이 계속 반복되고 회복되지 않는다.
  • Green 쪽 권한, 인증서, 네트워크 접근이 실패한다.
  • 전환이 끝났는데도 앱이 계속 Blue IP로 붙으려 한다.
07

이렇게 말하면 된다

소스 코드 설명 전에 이 스크립트로 전체 그림을 잡아주면 된다.

Blue/Green 전환은 DB 이사라고 보면 됩니다. Blue는 원래 쓰던 DB, Green은 새로 넘어갈 DB입니다. 문제는 DB는 새 집으로 이사했는데 애플리케이션의 주소록이 잠깐 예전 집을 가리킬 수 있다는 점입니다. 그래서 bg 플러그인은 JDBC 앞에서 교통정리를 합니다. 위험한 순간에는 잠깐 기다리게 하고, 전환 후 정리 구간에는 Green 쪽으로 연결을 바꿔줍니다. 이때 커넥션 풀이 잠깐 멈춘 것처럼 보일 수 있는데, 반드시 장애라는 뜻은 아닙니다. 전환 중 잘못된 DB로 쓰기 요청이 가는 것을 막기 위해 의도적으로 기다리는 구간일 수 있습니다.
질문이 나오면 이렇게 답하면 된다. “이 플러그인은 DB를 바꾸는 도구가 아니라, DB가 바뀌는 동안 앱의 JDBC 연결이 엉뚱한 곳으로 가지 않게 도와주는 도구입니다.”
